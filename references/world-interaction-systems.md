# 世界互動系統 Recipe

這份文件抽出 Stage 5-9 做的五個系統的**通用做法**，給下一個專案直接參考，不是這個遊戲的
變更紀錄（那個看 README 的 Stage Gate 表格）。每個系統都照同一個骨架整理：資料層、
state 遷移、核心邏輯、UI、美術需求、單元測試、E2E gate。

## 跨系統共通原則

在動手做任何一個系統前，先確認這幾條，這是本專案踩過坑換來的：

- **所有互動走「選動作 → 走到目標 → 播動作 → 結算」**，不要開一個不用走地圖就能觸發的
  全域按鈕當入口，哪怕只是「暫時方便測試」——這種捷徑很容易變成正式功能忘記拔掉。
- **解鎖條件要用真資源/真進度**，不能只是面板開關。判斷「能不能做 X」的函式要同時檢查
  等級/資源/故事旗標，不要只檢查其中一項就放行。
- **離線與上線行為要用同一套規則推導**，不要為離線寫特例（例如「離線固定給普通品質」
  這種語義不一致的捷徑，遲早會被抓到）。
- **新系統要跟既有系統的資料模型對照過一次**，尤其是「這個新系統的解鎖判斷，是不是
  依賴另一個系統之前的實作細節」——本專案真實發生過：把建築從「玩家建造才存在」改成
  「地圖常駐一律存在」之後，忘記同步更新動物解鎖的判斷邏輯，導致等級限制形同虛設，
  是後來一輪審核才抓到的迴歸。
- **每個免費、無成本的重複動作都要有冷卻或其他節流機制**，否則玩家會用「狂點」繞過
  你設計的節奏（本專案的照護系統一開始只有部分動作有冷卻，另一部分沒有，結果變成
  可以無成本狂點衝數值）。

## A. 世界解鎖區（Locked Region / Event Point）

**用途**：讓世界地圖有「看得到但進不去」的區域，配合資源/進度門檻解鎖，製造探索動機，
而不是一次把整張地圖攤開。

| 層面 | 做法 |
|---|---|
| 資料層 | 地圖磚標記所屬 region（例如 `tile.region = "east"`），配合 `state.flags`（布林開關，如 `bridgeRepaired`）記錄解鎖狀態；事件點用獨立設定（id/獎勵/是否一次性）而非硬編碼在互動函式裡 |
| state 遷移 | 若地圖維度變更，整個地圖連同 region/事件點重建；`flags` 物件要有預設值，migrate 時對舊存檔補齊 |
| 核心邏輯 | 「能否進入」的判斷（通常就是走路可達性函式，如 BFS 尋路的前置檢查）要同時檢查 region 解鎖狀態；解鎖動作的函式要检查資源是否足夠、故事進度是否達標，兩者都要，缺一個就擋下；事件觸發函式要處理「已觸發過」的冪等性（不能無限重複拿獎勵）|
| UI | 封鎖區要讓玩家「看得到但走不進去」（視覺降階：壓暗/加紋理，不要整個隱藏），解鎖當下要有明顯回饋（VFX/動畫），事件點要有地標式的美術而非空氣牆 |
| 美術需求 | 封鎖視覺通常可以純 CSS 處理（濾鏡/紋理疊圖），不用另外生圖；解鎖用的道具（例如橋）跟事件點地標（例如古樹）才需要額外素材 |
| 單元測試 | 解鎖前後的可達性（BFS）分別測試；解鎖函式的雙重門檻（資源不夠/進度不夠）各自獨立測試失敗案例；事件觸發的冪等性測試 |
| E2E gate | 封鎖前 BFS 不可達 → 解鎖動作消耗真資源 → 解鎖後 BFS 可達 → camera 能跟著角色進新區域 |

## B. NPC 對話系統

**用途**：讓世界「有人」，故事不再只靠側欄文字，同時可以承接教學/委託類任務觸發點。

| 層面 | 做法 |
|---|---|
| 資料層 | NPC 設定（id/顯示名稱/對應 sprite/依故事階段分組的台詞陣列）與地圖放置座標分開設定；`state.story` 加 `dialogueSeen`（是否見過）與 `dialogueLog`（對話記錄，給側欄用） |
| state 遷移 | 若地圖維度不變但新增 NPC 放置，要有「補放置」的重建邏輯（跟站點/建築同一套模式：檢查地圖上是否已有這個 NPC，沒有才補放） |
| 核心邏輯 | 「現在該講哪句台詞」的函式要**依當下故事進度現算**，不要存一個固定索引——玩家進度可能因為其他系統（補進度機制）跳過中間狀態，台詞階段要能跟著跳；同一階段有多句台詞時要有循環機制，不要每次都講第一句 |
| UI | 觸發方式是「走近」不是「點擊」——跟地圖上其他互動一致；同時呈現地圖泡泡（角色名+台詞，短暫顯示）跟側欄歷史記錄（可回顧） |
| 美術需求 | 每個 NPC 至少要 idle + talk 兩種姿態（各建議 2 幀做細微動態）；front-facing（面向玩家）比多方向省事，且符合「站在原地對話」的互動模型 |
| 單元測試 | 對話階段推導函式：給不同故事進度組合，驗證回傳的階段正確；對話觸發是否正確標記 `dialogueSeen` |
| E2E gate | 走近 NPC 出現對話泡泡且內容正確；同一個 NPC 在故事推進後，對話內容確實改變（不是卡在第一句）|

## C. 動物照護／親密度系統

**用途**：把「動物只是產物計時器」升級成需要互動維護的養成系統。

| 層面 | 做法 |
|---|---|
| 資料層 | 親密度用「上次照護當下的已收藏值 + 距今衰減」表示（存 `affinity` 數值 + `lastCaredAt` 時間戳），**不要用 tick 模擬**——這樣離線期間完全不需要特例處理，跟核心放置遊戲「用時間戳推導成長」的哲學一致。各照護動作各自的 `lastXAt` 欄位另外存，用來算冷卻與 UI 狀態圖示 |
| state 遷移 | 舊存檔的動物物件補齊新欄位（預設值：親密度 0、各 `lastXAt` 為 0 或建立時間），用 `Object.assign(預設值, 舊資料)` 模式確保新欄位一定存在但不覆蓋舊資料 |
| 核心邏輯 | 現算函式：`當下親密度 = min(上限, 已收藏值 - 距今時間 × 衰減率)`；不同照護動作要有不同的成本/效益權衡——**有實質成本的動作（消耗資源）可以用成本取代冷卻，但如果它還會覆蓋產出週期、直接出貨、或能被大量資源一次爆衝繞過節奏，還是要另外加冷卻或每日/每週期上限**；**沒有成本的動作（免費）一定要設冷卻**，否則會被無限次數的免費動作繞過設計意圖；品質分級要在「收集/照護當下」現算，不要鎖定某個歷史時間點的值；如果同一個動作會覆蓋某個時間戳欄位（例如餵食順便重置產出週期），要檢查是否有「已經產出但還沒收」的東西會被覆蓋吞掉，先自動收集再覆蓋 |
| UI | 親密度用進度條視覺化；狀態圖示只在「非理想狀態」才顯示（例如飢餓/口渴才顯示提示，已經是最佳狀態就不用額外標示），避免畫面過度雜亂；照護動作按鈕要反映冷卻狀態（disabled），且 UI 改變 state 後如果沒有走正常的按鈕點擊流程（例如測試腳本直接改 state），要記得手動觸發重繪，否則畫面停留在舊的 disabled 狀態 |
| 美術需求 | 每個照護動作對應的 VFX（至少 4-6 幀的簡單粒子效果）、非理想狀態圖示（每種狀態 1-2 幀）、動物在「理想狀態」下的專屬姿態（例如開心/進食），讓玩家從遠處就能看出動物現況 |
| 單元測試 | 親密度成長曲線（多次照護疊加）、衰減曲線（長時間不理會後歸零）、冷卻機制（連續呼叫同一動作要失敗）、離線結算是否套用跟上線同一套品質判斷、「已累積產物在下一次照護時是否被正確保留」的邊界案例 |
| E2E gate | 完整走一輪照護流程（走到動物旁 → 連續照護到理想狀態 → 收集到對應品質的產物），冷卻類行為在 E2E 裡用直接回撥時間戳模擬，不要真的等待 |

## D. 品質分級經濟

**用途**：讓同一種基礎產出依玩家投入程度分出高低價值，不需要開發全新的產出種類。

| 層面 | 做法 |
|---|---|
| 資料層 | 品質分級的 item 定義用**程式生成**（迴圈跑過每個基礎品項 × 每個品質等級），不要手動列舉每個組合——組合數會隨基礎品項數量倍增，手動維護容易漏改 |
| state 遷移 | 舊存檔的「已發現／已收集」記錄（例如 `collected` map）如果原本只有基礎品項 id，遇到新加入的品質分級 id 要能安全 fallback（視為未發現，不是報錯或當成已發現）；舊 inventory 裡不存在的品質 id 要有預設值 0，不能讀到 `undefined` |
| 核心邏輯 | 判斷「這個品項現在該不該進入市場出口（訂單池/商店）」時，沿用「玩家自己發現過才算數」的閥門模式：不要讓玩家還沒摸過某個系統，就被隨機生成的內容要求交付對應產物——這通常表示「已收集數量 > 0」才把該 item id 加進可用清單 |
| UI | 品質分級要有清楚的視覺區分（圖示/色框/星等），不能只顯示 raw item id 或文字後綴；訂單/倉庫列表要能依基礎品項分組，讓玩家一眼看出「同一種東西的三個品質版本」而不是三個無關項目；玩家還沒發現的品質分級不要在 UI 提前預告（會破壞「發現感」）|
| 美術需求 | 每個品質分級至少要有可辨識的圖示差異（例如疊加星光/色框，不必整張圖重畫）；如果分級數固定（如三級），可以用同一個 icon 模板配不同疊加層，不必每個基礎品項各畫三張完整圖 |
| 單元測試 | 品質判斷函式（給定某個數值，回傳對應等級）的邊界值測試；訂單池閥門（未發現/已發現）測試；售價計算測試；舊存檔遷移後品質 id 讀取不炸掉的測試 |
| E2E gate | 通常搭配 C 的照護流程一起驗證（照護到位 → 收集高品質產物 → 賣出/交付驗證經濟閉環）|

## E. 世界狀態視覺化（天氣／時段等）

**用途**：把已經存在的數值狀態（天氣、日夜、季節……）從「只有一個 HUD 圖示」升級成
「玩家在世界本身就能感受到」，成本低但每次登入的觀感提升很直接。這通常是把一個已經
上線很久、只影響數值但沒有對應視覺的系統「補後腳」，而不是從零做一個新系統。

| 層面 | 做法 |
|---|---|
| 資料層 | 通常不需要新資料——如果數值系統（天氣/日夜）已經存在，直接讀現有的 `state` 欄位即可，不要為了視覺效果另外存一份重複狀態 |
| state 遷移 | 通常不需要，純視覺層對存檔格式沒有新增欄位 |
| 核心邏輯 | 通常不需要改動——如果現有的「現在是什麼狀態」判斷函式已經是純函式（吃 `state, now`），視覺層直接呼叫它即可，不要另外複製一份判斷邏輯到 UI 層 |
| UI | 新增一個獨立的世界層濾鏡（overlay），疊在整個視口上而不是疊在會被 camera 平移的世界容器裡——否則濾鏡會跟著世界滾動，看起來像貼圖而不是天氣；只在狀態真的改變時才切換 class，不要每個 tick 都重設，否則 CSS 動畫會一直被打斷重播；狀態掛 `data-audit`/`data-<狀態>` 屬性給外部稽核 |
| 美術需求 | 優先考慮純 CSS（漸層、`repeating-linear-gradient`、`mix-blend-mode`）能不能做到想要的效果——世界狀態濾鏡通常不需要新的 sprite/gpt-image-2 素材，純 CSS 風險更低、也不用花 API 額度；只有效果真的需要具體圖案（例如飄落的樹葉貼圖）才考慮補素材 |
| 單元測試 | 通常不需要新測試——如果核心判斷邏輯本來就有測試覆蓋，純視覺層改動不需要重複測 |
| E2E gate | 強制切換到每個狀態（不要等真實隨機觸發），斷言視覺層的 class/data 屬性正確跟隨；斷言原本的數值效果（例如天氣影響售價/成長速度）沒有被這次改動動到；桌機/手機都要確認新的視覺層沒有造成水平溢出 |
