# Stage Gate Playbook

這份文件講的是**方法**，不是這個遊戲的變更紀錄。目的是讓你在別的專案也能複製同一套節奏：
一階段一上線、每階段可玩可測可上線、多 agent 協作時先驗證再動手。

想看「這個遊戲每個 Stage 實際做了什麼」，去讀 README 的 Stage Gate 表格與各 `production-directive-*.md`；
這份文件只講「下一次你自己開一個新專案，要怎麼跑這個流程」。

## 什麼是 Stage Gate

一個 Stage = 一個**獨立可上線**的功能切片。判斷「這算不算一個 Stage」的標準：

- 玩家能感受到具體變化（不是重構、不是預備工作）
- 有明確的驗收條件（gate），不是「看起來不錯就好」
- 做完當下就能 `commit → push → CI 綠燈 → 線上可玩`，不留著跟下一批一起上

**反例**：「先把資料模型都改好，UI 之後再接」不是一個 Stage，那是半成品。資料模型跟 UI
要在同一個 Stage 裡一起交付，因為沒有 UI 就沒辦法驗收「玩家能不能感受到」。

## 每個 Stage 的標準循環

```
1. 定範圍       —— 這個 Stage 具體要讓玩家做到什麼新的事？寫成一兩句話的驗收描述。
2. 資料模型     —— config/state 先設計好欄位與預設值，想清楚舊存檔怎麼補（migrate）。
3. 核心邏輯     —— 純函式、無 DOM、吃 (state, now, ...) 回傳結果，不碰 setTimeout/Date.now()
                   內部呼叫（測試才能注入固定時間點）。
4. 互動/UI      —— 地圖路由、側欄面板、CSS。原則：「選動作 → 走到目標 → 播動作 → 結算」，
                   不要用可以繞過地圖的全域按鈕當早期唯一入口。
5. 美術（若需要）—— 見 art-pipeline-v4.md：先寫 art-config 規格 → 生圖 → 切割 → 驗證，
                   四步都做完才進下一步，不要素材生了就先擱著。
6. 測試         —— 先補 Node 單元測試（core logic，快、可精準驗證數值），
                   再補/延伸 E2E（見 e2e-gate-checklist.md，桌機+手機都要跑）。
7. 瀏覽器實測    —— 真的開瀏覽器看一次，尤其是視覺類改動；螢幕截圖比看程式碼可靠。
8. 上線         —— commit（訊息要講清楚「為什麼」不是「做了什麼」）→ push → 盯著 CI 跑到綠燈
                   → 確認 GitHub Pages 真的部署新版本。
```

不要跳步。最常見的失敗模式是「先把好幾個 Stage 的程式碼都寫完，最後一次測試」——
一旦出錯，你不知道是哪個改動造成的，而且中間任何一步卡住，前面的工作都無法上線驗收。

## 次版本號（X.5、X.1）什麼時候用

一個 Stage 上線後，如果馬上發現需要修的問題（迴歸、平衡、邊界案例），**不要**把它算進下一個
Stage 的範圍，也不要重新編號——開一個 `Stage X.1`（或 `.5`，看你專案的慣例）獨立處理、獨立上線。

理由：
- 保持每個 commit 的意圖單一，方便之後回溯「這個修正是為了什麼」
- 讓下一個整數 Stage 的範圍維持乾淨，不要把補丁寫進新功能的方法論裡
- 這也是 Stage Gate 表格能一直保持「每列都有明確 gate」的原因——補丁本身也要有 gate

這個專案的例子：Stage 4.5（修任務完成度顯示 bug + 改名）、Stage 6.5（修 Stage 4 改地圖模型
留下的動物解鎖/訂單經濟迴歸）、Stage 7.1（修 Stage 7 上線後發現的照護系統邊界問題）。

## 多 agent 協作：分工與「先驗證再動手」

如果你有兩個以上的 AI agent 協作（例如一個負責審核/企劃/美術規格、一個負責實作），
最重要的一條規則是：**審核端的每一個具體主張，實作端都要自己讀程式碼驗證過，才能動手修**。

### 為什麼要驗證，不能直接信

審核端（不管是人類還是另一個 agent）給的意見，品質差異很大：

- 可能是真的抓到 bug（例如：「Lv1 能買到 Lv5 才該解鎖的動物」——這種可以直接用測試腳本
  或 `grep` 驗證：實際跑一次 `buyAnimal()`，看它是否真的沒檢查等級）
- 可能是誤判（審核端沒看到某個已存在的防護機制）
- 可能是設計選擇，不是 bug（例如：「訂單價格不受天氣影響」可能是刻意設計）

**分類方式**：
1. 能用程式碼證據（讀原始碼 / 跑一段驗證腳本）直接證實或推翻的 → 驗證後直接處理
2. 涉及數值平衡、UX 偏好的主觀判斷 → 標記出來，讓人類做決定
3. 審核端自己也不確定的（例如：「這可能是刻意設計，需要確認」）→ 一樣交給人類

在這個專案裡，Codex 兩輪審核總共提出 9 個具體問題，全部先用 `grep`/`node -e` 驗證程式碼
行為後才動手修——過程中還發現了審核端沒提到的 bug（玩家自建的第二個建築因為缺欄位，
動物永遠不會畫在地圖上），這也是「你自己動手驗證」的附帶好處：驗證的過程會讓你重新
理解整段程式碼，順帶抓到審核端沒看到的問題。

### 分工建議

| 角色 | 負責 |
|---|---|
| 人類 | 最終設計意圖的仲裁（例如數值平衡方向）、是否花費真實資源（API 額度）的授權、排程優先順序 |
| 審核/企劃/美術 agent | 讀 repo 現況給獨立意見（不要先餵它摘要，讓它自己讀，才會有獨立判斷）、規劃路線圖、
  草擬美術規格（格式要能直接餵給生圖 pipeline，不要只給文字描述） |
| 實作 agent | 驗證審核端的每個主張、決定技術細節、寫程式碼、補測試、跑 gate、上線 |

### 回合節奏

一輪協作大致長這樣：

```
實作端：完成一個 Stage，簡述交付內容（不要只丟連結，把關鍵設計決策寫清楚）
    ↓
審核端：讀 repo 現況（不是讀摘要），給具體、可驗證的意見（不要只給「感覺不錯」）
    ↓
實作端：逐項驗證（讀程式碼/跑腳本），標出「確認屬實」vs「需要人類決定」
    ↓
人類：對需要決定的項目給方向
    ↓
實作端：修正、補測試、跑全部 gate、上線
    ↓
（重複，進入下一輪）
```

每一輪都要收斂成一次實際上線（commit + push + CI 綠燈），不要讓討論無限延伸而不產出。

## 常見失敗模式

- **只交概念稿**：規劃文件寫得再詳細，沒有可玩、可測、可上線的東西都不算完成一個 Stage。
- **測試補在最後**：先寫測試（至少是核心邏輯的 Node 測試）再實作互動層，抓 bug 成本低很多。
- **改了行為忘記回頭補舊測試**：任何會改變既有函式輸出/副作用的修正（例如幫某個動作加冷卻），
  一定要重跑全部既有測試，看有沒有測試的假設被打破（這個專案至少發生過 3 次）。
- **沒有實際開瀏覽器看**：視覺類改動只看 E2E 綠燈不夠，E2E 檢查的是「有沒有這個 DOM 元素」，
  不是「畫面好不好看」，兩者都要做。
- **忽略 CI 的執行環境跟本機不同**：CI 常常沒有本機瀏覽器/字型，涉及像素檢查的驗證器要能
  在缺瀏覽器時優雅降級（warn 而非直接失敗），但本機開發時要確保它有跑到完整檢查。
